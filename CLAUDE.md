## Configuration Files

- `.mcp.json` — MCP server configurations. Each entry maps a service name to its server command and environment variables.
- `.env` — Contains all API keys and secrets. **Never commit this file.** Never print its contents to output.
- `skills/` — Contains skill files (`.md`) with instructions for specific workflows. Each skill is tagged with the platforms it requires.
- `.env.example` — A template showing what environment variables are needed. Use this to help the user set up missing keys.

## Operating Principles

### 1. Always Check Connectivity First
Before performing any action on a service, verify the MCP server is responding:
```bash
bash scripts/verify-mcp.sh
```

### 2. Use the Right Tool for the Job
- Use MCP servers for service-specific operations (creating issues, querying databases, sending emails)
- Use skills from `skills/` for multi-step cross-platform workflows
- Use Context7 to look up current API docs when you're unsure about a service's API shape
- Use OpenRouter for any AI-powered analysis or generation tasks

### 3. Handle Missing Credentials Gracefully
If an MCP server fails to connect, check `.env` for the relevant API key. If the key is missing or invalid:
1. Tell the user exactly which key is needed and for which service
2. Provide the signup URL or settings page where the key can be found
3. Offer to resume once they've added it

### 4. Security
- **Never** print API keys, tokens, or secrets to output
- **Never** commit `.env` to git
- **Always** use environment variables, never hardcode credentials
- When creating resources, use the principle of least privilege

### 5. Idempotency
Before creating a resource, check if it already exists. For example:
- Before creating a Linear team, check if one with that name exists
- Before creating a Supabase table, check if it already exists
- Before creating a Resend API key, check if one with that name exists

## Common Workflows

### "Set up my project tracking"
1. Use Linear MCP to create a workspace (if not exists)
2. Create teams for Engineering, Design, Product
3. Create a backlog project
4. Report the Linear workspace URL to the user

### "Create a landing page signup form"
1. Use Supabase MCP to create a `waitlist` table with columns: email, created_at, source
2. Create an RLS policy allowing public inserts
3. Use Resend MCP to create a welcome email template
4. Provide the user with the Supabase API URL and anon key for their frontend

### "Sync my Shopify orders to Xero"
1. Use the `shopify-orders-to-xero` skill from `skills/cross-platform/`
2. Fetch recent orders from Shopify MCP
3. Map each order to a Xero invoice structure
4. Create draft invoices in Xero MCP
5. Report a summary of what was synced

## Error Handling

When an MCP server call fails:
1. Check if the error is authentication-related → guide the user to refresh their token
2. Check if the error is rate-limiting → implement exponential backoff or queue the request
3. Check if the error is a missing resource → create it if appropriate, or inform the user
4. For unknown errors → log the error, inform the user, and suggest next steps

## OpenRouter Usage

The project is configured to use OpenRouter's free endpoint (`openrouter/free`). This provides access to free models with a 200K token context window at $0/token.

If a task requires a specific model capability (e.g., vision, structured output), OpenRouter's free router automatically selects a model that supports those features.

If you hit rate limits on the free tier, inform the user and suggest they add credits to their OpenRouter account for paid model access.
